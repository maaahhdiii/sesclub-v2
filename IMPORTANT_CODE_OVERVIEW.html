SES'Club Important Code Overview

This version shows code segments from the most important files, then explains what each segment does and why it matters.

1. JWT Authentication

This is the main login system for API requests.

sesclubs/settings/base.py + users/serializers.py
REST_FRAMEWORK = {
      "DEFAULT_AUTHENTICATION_CLASSES": (
        "rest_framework_simplejwt.authentication.JWTAuthentication",
      ),
      "DEFAULT_PERMISSION_CLASSES": (
        "users.permissions.IsVerified",
      ),
    }

    SIMPLE_JWT = {
      "ACCESS_TOKEN_LIFETIME": ACCESS_TOKEN_LIFETIME,
      "REFRESH_TOKEN_LIFETIME": REFRESH_TOKEN_LIFETIME,
      "ROTATE_REFRESH_TOKENS": True,
      "BLACKLIST_AFTER_ROTATION": True,
      "UPDATE_LAST_LOGIN": True,
      "ALGORITHM": "HS256",
    }

    class CustomTokenObtainPairSerializer(TokenObtainPairSerializer):
      def validate(self, attrs):
        email = attrs.get("email")
        password = attrs.get("password")
        user = authenticate(username=email, password=password)
        if user is None:
          user = authenticate_club_portal(email, password)
        if user is None:
          raise serializers.ValidationError("Invalid credentials")
        data = super().validate(attrs)
        data["role"] = user.role
        data["selected_club_id"] = resolve_selected_club(user)
        return data
This flow has two important layers. First, settings force JWT auth and verified-user permissions across the API. Second, the custom login serializer accepts either a normal account or a club-portal credential, then returns token data plus role and selected club context. That means one login endpoint can power both the main platform and club portal without duplicating auth logic.

2. Google Login Routing

This decides where a user goes after signing in with Google.

users/views.py
class GoogleClubLoginView(SocialLoginView):
      def get(self, request, *args, **kwargs):
        request.session["google_portal_target"] = "club"
        club_id = request.GET.get("club_id")
        if club_id:
          request.session["club_id"] = club_id
        return super().get(request, *args, **kwargs)


    class GoogleStudentLoginView(SocialLoginView):
      def get(self, request, *args, **kwargs):
        request.session["google_portal_target"] = "student"
        return super().get(request, *args, **kwargs)


    class GoogleSuccessView(APIView):
      def get(self, request, *args, **kwargs):
        target = request.session.get("google_portal_target")
        club_id = request.session.get("club_id")
        user = request.user
        if target == "club" and club_id:
          return redirect(f"/club/?club_id={club_id}")
        if user.is_administrator:
          return redirect("/admin-page/")
        if user.is_club_manager or is_club_member(user):
          return redirect("/club/")
        return redirect("/student/")
The critical fix here is context preservation. The app stores both intent and club ID before leaving for Google OAuth, then reads them on return to route correctly. Without this, users can authenticate successfully but land in the wrong portal. The final redirect block also applies role-based fallback logic, so admins, club users, and students each get sent to the right destination.

3. Club Membership Model

This stores who belongs to a club and what role they have inside it.

clubs/models.py
class Club(models.Model):
    name = models.CharField(max_length=120, unique=True)
    president = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        null=True,
        blank=True,
        related_name="president_of",
        on_delete=models.SET_NULL,
    )


class ClubMembership(models.Model):
    ROLE_CHOICES = (
        ("member", "Member"),
        ("vice_president", "Vice President"),
        ("president", "President"),
    )
    STATUS_CHOICES = (
        ("active", "Active"),
        ("pending", "Pending"),
        ("left", "Left"),
    )

    club = models.ForeignKey(Club, related_name="memberships", on_delete=models.CASCADE)
    user = models.ForeignKey(settings.AUTH_USER_MODEL, related_name="club_memberships", on_delete=models.CASCADE)
    internal_role = models.CharField(max_length=32, choices=ROLE_CHOICES, default="member")
    status = models.CharField(max_length=32, choices=STATUS_CHOICES, default="active")

    class Meta:
        unique_together = ("club", "user")
This model captures both identity and state of a person inside a club. The role field controls authority inside the portal, while the status field controls lifecycle transitions like pending, active, or left. The unique constraint prevents duplicate membership records for the same user in the same club, which protects downstream logic in permissions, requests, and dashboard counts.

4. Membership Management

This is the workflow that adds or removes members and sends emails.

clubs/views.py
class ClubMembershipViewSet(ModelViewSet):
    queryset = ClubMembership.objects.select_related("club", "user")
    serializer_class = ClubMembershipSerializer
    permission_classes = [CanManageClub]

    def get_queryset(self):
        qs = super().get_queryset()
        club_id = self.request.query_params.get("club")
        if club_id:
            qs = qs.filter(club_id=club_id)
        return qs

    def perform_create(self, serializer):
        membership = serializer.save()
        self._send_member_welcome_email(membership.user, membership.club)
        if membership.internal_role == "president":
            self._send_club_manager_email(membership.user, membership.club)

    def perform_destroy(self, instance):
        user, club = instance.user, instance.club
        instance.delete()
        AuditLog.log("member_removed", user=user, club=club)
This is the operational part of membership management. Query filtering keeps list APIs scoped to one club, create hooks send onboarding messages, and special-role creation can trigger manager credential emails. Deletion runs through one controlled path so audit logging and follow-up side effects remain consistent instead of being scattered across frontend calls.

5. Profile Data for the Frontend

This is how the frontend learns the current role and membership state.

users/views.py
class ProfileView(RetrieveAPIView):
    serializer_class = UserSerializer
    permission_classes = [IsAuthenticated]

    def retrieve(self, request, *args, **kwargs):
        user = request.user
        selected_club_id = request.query_params.get("club_id")
        membership = ClubMembership.objects.filter(
            user=user,
            club_id=selected_club_id,
            status="active",
        ).first()

        payload = {
            "id": user.id,
            "email": user.email,
            "role": user.role,
            "is_club_manager": user.is_club_manager,
            "is_club_member": membership is not None,
            "club_internal_role": membership.internal_role if membership else None,
            "club_membership_status": membership.status if membership else None,
        }
        return Response(payload)
This endpoint is the frontend truth source after authentication. It merges global identity (user role) with club-specific identity (membership role and status) based on the selected club. Because this response is explicit, the UI can render correct nav visibility, badge labels, and access checks without guessing from incomplete token data.

6. Settings That Control Everything

This file ties authentication, email, OAuth, and API behavior together.

sesclubs/settings/base.py
AUTH_USER_MODEL = "users.User"

AUTHENTICATION_BACKENDS = (
    "social_core.backends.google.GoogleOAuth2",
    "django.contrib.auth.backends.ModelBackend",
)

SOCIAL_AUTH_GOOGLE_OAUTH2_KEY = env("GOOGLE_CLIENT_ID", default="")
SOCIAL_AUTH_GOOGLE_OAUTH2_SECRET = env("GOOGLE_CLIENT_SECRET", default="")

SOCIAL_AUTH_PIPELINE = (
    "social_core.pipeline.social_auth.social_details",
    "social_core.pipeline.social_auth.social_uid",
    "social_core.pipeline.social_auth.auth_allowed",
    "social_core.pipeline.social_auth.social_user",
    "social_core.pipeline.user.get_username",
    "social_core.pipeline.user.create_user",
    "users.pipeline.validate_sesame_domain",
    "users.pipeline.enforce_role_rules",
    "social_core.pipeline.social_auth.associate_user",
    "social_core.pipeline.social_auth.load_extra_data",
)

EMAIL_BACKEND = "django.core.mail.backends.smtp.EmailBackend"
DEFAULT_FROM_EMAIL = env("DEFAULT_FROM_EMAIL", default="noreply@sesclub.local")
This settings block wires together the custom user model, Google OAuth backend, post-login pipeline checks, and outbound email behavior. In practice, this is the highest-impact configuration surface in the app: if pipeline order is wrong, domain checks can be bypassed; if auth backends are wrong, social login fails; if email settings are wrong, verification and manager credential flows silently break.
